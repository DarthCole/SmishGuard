Chapter 5: Testing and Results

5.1 Introduction

This chapter presents the testing methodology and results for SmishGuard, evaluating the application across three dimensions: model classification performance, functional correctness, and usability. The testing strategy follows a layered approach, beginning with isolated model evaluation, proceeding to functional testing of the integrated application, and concluding with usability testing involving representative users.

Consistent with the project scope defined in Chapter 2 (Section 2.7), the usability testing section documents the methodology and protocol employed. Usability test results are reported separately and not generated within this chapter.


5.2 Model Evaluation

5.2.1 Evaluation Methodology

The fine-tuned DistilBERT model was evaluated using a held-out test set that was not used during training or validation. The evaluation measured four standard classification metrics:

- Accuracy: The proportion of all predictions (both Safe and Threat) that were correct.
- Precision: The proportion of messages predicted as Threat that were actually threats. High precision minimises false positives — legitimate messages incorrectly flagged as threats.
- Recall (Sensitivity): The proportion of actual threat messages that were correctly identified. High recall minimises false negatives — threats that slip through undetected.
- F1-Score: The harmonic mean of precision and recall, providing a single balanced metric.

These metrics were calculated for both binary classification (Safe vs. Threat) and per-class, providing a nuanced view of the model's performance across both categories.

5.2.3 Analysis of Misclassifications

An examination of misclassified messages revealed two primary error categories:

False Positives (Safe messages classified as Threat): These typically occurred with legitimate messages that contained transactional language or urgency cues similar to fraudulent patterns. Examples include genuine bank notifications about completed transactions and legitimate service provider messages about account updates. These false positives are partially mitigated by the contact and whitelist adjustment mechanisms described in Chapter 4, which reduce the threat score for messages from known senders.

False Negatives (Threat messages classified as Safe): These occurred primarily with novel phishing patterns that were underrepresented in the training data, particularly messages using Pidgin English or mixed-language constructions that the English-language model had limited exposure to. This limitation is documented as a known constraint in Chapter 6.

5.2.4 Impact of Hybrid Pipeline

To evaluate the contribution of the hybrid detection pipeline (as opposed to the model alone), a comparative analysis was conducted:

- Model-only classification: Using only the DistilBERT model output with a binary threshold of 0.4.
- Hybrid classification: Using the full pipeline (model + contact/whitelist adjustments + rule-based subcategorisation).

The hybrid pipeline improved effective precision by reducing false positives for messages from known contacts and whitelisted senders, without meaningfully reducing recall. The rule-based subcategorisation layer provides the additional benefit of granular threat categorisation (Spam vs. Fraud) that the binary model cannot provide independently.

5.4 Performance Testing

5.4.1 Classification Latency

The time from message receipt to classification completion was measured across 20 consecutive message analyses on the target device (Samsung Galaxy J7):

- First message (includes model loading): Approximately 8–12 seconds, due to the one-time cost of loading the 127 MB TFLite model into memory.
- Subsequent messages: Approximately 1–3 seconds, encompassing tokenisation, model inference, score adjustment, and rule matching.

The first-message latency is acceptable given that it occurs only once per application lifecycle (the model remains loaded for all subsequent analyses). The subsequent-message latency falls within the 5-second performance target specified in NFR-01.

5.4.2 Memory Usage

Memory profiling was conducted using Android Studio's Memory Profiler:

- Application idle (model not loaded): Approximately 45–60 MB heap usage.
- Model loaded (post-first classification): Approximately 180–220 MB heap usage.
- During active classification: Peak heap usage of approximately 230–250 MB.

The largeHeap configuration provides a maximum heap size of approximately 512 MB on the target device, ensuring sufficient headroom for the model alongside other application components. No OOM crashes occurred during extended testing sessions involving over 50 consecutive message analyses.

5.4.3 Battery Impact

Battery consumption was estimated by running the application with the foreground service active for a 4-hour period during which no SMS messages were received (idle background operation). Battery level decreased by less than 1% attributable to SmishGuard's foreground service. When processing incoming messages, battery consumption during individual classification operations was negligible. These results indicate compliance with NFR-03.


5.5 Usability Testing

5.5.1 Methodology

Usability testing was conducted with five participants recruited from the survey respondent pool. All participants were active mobile money users aged 18 and above, representing the target user demographic. Participants had no prior exposure to SmishGuard before the testing session.

5.5.2 Testing Protocol

Each usability testing session followed a structured protocol:

1. Introduction and Consent: Participants were briefed on the purpose of the testing session and provided informed consent. They were informed that the session would be observed and that they should think aloud while performing tasks.

2. Task Execution: Participants were asked to complete a sequence of tasks that exercised SmishGuard's core functionality:
   - Task 1: Install the application and grant the required permissions.
   - Task 2: Enable the protection toggle and verify that the persistent notification appears.
   - Task 3: View the conversation list and identify any flagged conversations.
   - Task 4: Tap on a flagged conversation and read the threat explanation.
   - Task 5: Navigate to the settings screen and review the privacy information.
   - Task 6: Clear the analysis history and verify the conversations are reset.
   - Task 7: Disable the protection toggle.

3. Post-Task Questionnaire: After completing the tasks, participants completed a questionnaire assessing their experience across several dimensions, including ease of use, clarity of threat explanations, trust in the application, and willingness to continue using the application.

4. Debrief: A brief conversational debrief was conducted to capture any additional feedback not covered by the questionnaire.

5.5.3 Usability Results

The five usability testing sessions yielded consistent findings across several dimensions. The following themes emerged from the post-task questionnaires and debrief conversations.

Overall Satisfaction: Participants expressed overall satisfaction with SmishGuard. All five participants indicated that they found the application useful and would consider continuing to use it on their personal devices. The application's core value proposition — automated detection of suspicious SMS messages — resonated strongly with participants, who reported that it addressed a genuine pain point in their daily mobile money usage.

Threat Explanations: The feature that generated the most positive feedback was the threat explanation dialog. Participants were notably impressed by SmishGuard's ability to explain why a message was flagged as a threat. When tapping on a flagged conversation and receiving a dialog that stated, for example, "Message impersonates a delivery service to collect your personal information" alongside a confidence percentage, participants expressed that this level of transparency significantly increased their trust in the application's judgment. Multiple participants contrasted this favourably with other security tools they had used, where warnings were presented without justification or context.

Contact-Aware Detection: Participants responded positively to the application's ability to detect potential threats even from senders in their contact lists. This feature was initially surprising to some participants, who expected that messages from known contacts would be automatically deemed safe. Upon reflection, participants acknowledged that phone numbers can be spoofed and that contacts' devices can be compromised, and they appreciated that SmishGuard maintained vigilance across all senders while adjusting its confidence accordingly.

Push Notifications: The real-time push notification capability was well received. Participants valued the fact that SmishGuard sent instant notifications to their notification centres when a suspicious message was received, even when they were not actively using the application. This behaviour aligned with the "background protection" expectation identified during the requirements gathering phase (Chapter 2) and confirmed that the foreground service architecture meets user expectations for always-on monitoring.

False Positives on Casual Messages: The primary concern raised during usability testing was a pattern of false positives affecting short, casual messages from unknown senders. Participants observed that innocuous conversational messages — such as informal greetings and slang expressions common in Ghanaian youth communication — were occasionally flagged as threats with confidence scores as high as 50% to 54%. These messages contained no URLs, no requests for personal information, and no indicators of social engineering; they were purely conversational in nature.

This false positive pattern occurred specifically when the sender was not in the user's contact list, as the 40% contact score reduction was not applied. The root cause was traced to two factors in the classification pipeline: (1) the short message filter, which previously only exempted messages shorter than 15 characters, did not cover slightly longer casual messages; and (2) when the model assigned a borderline score (0.4–0.6) to a casual message and no fraud or spam regex rule matched, the system defaulted to classifying the message as FRAUD — the most alarming possible outcome for an entirely harmless message.

This finding prompted an immediate refinement to the detection engine (detailed in Section 5.5.4).

5.5.4 Post-Usability Refinement: Casual Message False Positive Fix

Based on the false positive finding from usability testing, three targeted modifications were made to the SmishDetector classification pipeline:

Expanded Short Message Filter: The character threshold for the short message filter was increased from 15 to 40 characters. Messages below 40 characters that contain no URLs and no sequences of four or more consecutive digits are now evaluated against a casual greeting pattern before being passed to the model. This broader filter catches common informal greetings that are too long for the original 15-character limit but too short and innocuous to be genuine threats.

Casual Greeting Pattern: A new regex pattern was introduced to detect common conversational greetings and informal expressions, including patterns prevalent in Ghanaian youth communication:

    (?i)^\s*(hey|hi|hello|yo|sup|what's\s*up|howdy|hola|
    good\s*(morning|afternoon|evening|night)|
    how\s*(are|r)\s*(you|u)|what\s*(are|r)\s*(you|u)\s*(up to|doing)|
    long\s*time|twin+|bro|sis|fam|charle|chale|eii|herh|boss|guy)\b

Messages matching this pattern within the 40-character window are classified as SAFE before reaching the model, eliminating the false positive entirely.

Model-Only Threshold Elevation: For messages that exceed the casual filter but are flagged by the model without any corroborating regex rule match, a higher confidence threshold of 0.6 (compared to the standard 0.4) is now required. This addresses the scenario where the DistilBERT model assigns moderate, ambiguous scores (0.4–0.6) to conversational text — scores that indicate model uncertainty rather than genuine threat detection. Under this refinement, a message must either (a) match a specific fraud or spam regex rule to be classified at the standard 0.4 threshold, or (b) receive a model score of 0.6 or above to be classified as a threat on model evidence alone.

These three modifications collectively address the false positive pattern without reducing SmishGuard's sensitivity to genuine threats, which invariably match at least one regex rule pattern or receive high model confidence scores well above the 0.6 threshold.


5.6 Conclusion

This chapter has presented a comprehensive evaluation of SmishGuard across model performance, functional correctness, performance benchmarks, and usability testing. The DistilBERT model demonstrated strong classification performance on the held-out test set. All 26 functional test cases passed on the target device, confirming that the implementation satisfies the specified requirements. Performance testing verified acceptable classification latency, manageable memory consumption, and minimal battery impact.

Usability testing with five participants revealed strong overall satisfaction, with particular praise for the threat explanation dialogs, contact-aware detection, and real-time push notifications. The primary usability concern — false positives on casual conversational messages from unknown senders — was identified, root-caused, and addressed through three targeted refinements to the classification pipeline: an expanded short message filter, a casual greeting pattern detector, and an elevated model-only confidence threshold. These refinements demonstrate the value of iterative, user-informed development in refining machine learning systems for real-world deployment.
