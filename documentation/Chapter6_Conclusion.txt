Chapter 6: Conclusion and Recommendations

6.1 Summary

This project set out to address the growing threat of SMS-based fraud (smishing) targeting mobile money users in Ghana, a population that is increasingly reliant on mobile financial services yet disproportionately underserved by existing fraud detection technologies. The three research objectives articulated in Chapter 1 guided the project:

1. To design and implement a hybrid SMS fraud detection system that combines machine learning with rule-based heuristics for accurate, granular threat classification.

2. To develop an on-device detection architecture that preserves user privacy by processing all SMS data locally, without transmitting information to external servers.

3. To evaluate the detection system's effectiveness and usability with representative Ghanaian mobile money users.

SmishGuard, the resulting application, addresses all three objectives. The system employs a fine-tuned DistilBERT transformer model, exported as a TensorFlow Lite artefact for on-device inference, combined with a contextual score adjustment mechanism (contact and whitelist awareness) and a 16-pattern regex rule engine that subcategorises threats into Spam and Fraud. This hybrid pipeline achieves the granularity that purely model-based approaches lack and the generalisation that purely rule-based approaches cannot provide.

The architectural design prioritises privacy through multiple reinforcing mechanisms: all processing occurs on-device, encrypted storage protects persisted data, network security configurations block cleartext traffic, and the application stores only classification metadata — never SMS content. The foreground service architecture ensures real-time protection, classifying incoming messages within seconds of arrival and displaying informative notifications for detected threats.

The application was tested on a Samsung Galaxy J7 running Android 8.1 — a mid-range device representative of the Ghanaian market. All 26 functional test cases passed, classification latency remained within acceptable bounds, and memory consumption remained sustainable with the largeHeap configuration. The user interface provides intuitive visual indicators (colour-coded dots and backgrounds) and human-readable threat explanations that address the explainability needs identified during stakeholder engagement.


6.2 Addressing the Research Questions

This section revisits the research questions posed in Chapter 1 and summarises how the project's findings address each one.

Research Question 1: How can machine learning techniques be effectively applied to detect SMS-based fraud in the Ghanaian mobile money context?

The project demonstrates that a fine-tuned DistilBERT model, trained on a dataset containing Ghana-specific SMS samples, can effectively distinguish between legitimate and fraudulent SMS messages. The model's statistical generalisation handles novel phishing patterns that do not match predefined rules, while the rule-based component provides targeted detection for known Ghana-specific fraud vectors (e.g., GhanaPost impersonation, USSD lottery scams, mobile money verification demands). The hybrid approach addresses the research question by showing that combining the two techniques yields superior results to either approach in isolation: the model provides breadth, and the rules provide depth.

Research Question 2: What architectural approach can enable on-device SMS fraud detection that preserves user privacy while maintaining acceptable performance on mid-range Android devices?

The on-device architecture deployed in SmishGuard demonstrates that privacy-preserving SMS fraud detection is technically feasible on mid-range Android hardware. The key enablers are: DistilBERT's efficiency relative to full-size BERT (60% faster, 60% smaller), TensorFlow Lite's optimised mobile inference engine, float16 quantisation for model size reduction, and the lazy loading singleton pattern for memory management. The architecture eliminates the need for server-side processing, ensuring that no SMS data leaves the device boundary. Performance testing confirmed sub-3-second classification latency for sustained operations and sustainable memory and battery consumption.

Research Question 3: How do Ghanaian mobile money users perceive and interact with an SMS fraud detection application, and what features do they consider most important?

Stakeholder engagement through surveys (88 participants) and semi-structured interviews (5 participants) revealed that Ghanaian mobile money users experience frequent exposure to SMS fraud, desire automated background detection, prioritise privacy assurances (particularly on-device processing commitments), and value human-readable explanations for flagged messages. These findings informed the application's design at every level: the foreground service provides always-on background detection; the privacy architecture implements the on-device commitment; and the explanation dialog translates technical classification results into actionable user information. Usability testing with five participants provided direct feedback on whether these design decisions resonate with users in practice.


6.3 Contributions

This project makes the following contributions:

1. A Hybrid Detection Pipeline for SMS Fraud: SmishGuard introduces a multi-stage classification approach that combines transformer-based inference, contextual score adjustment, and rule-based subcategorisation. This pipeline provides three-category classification (Safe, Spam, Fraud) with explainability — a capability not offered by existing commercial tools such as Google Messages or Truecaller, which provide binary spam filtering without explanatory detail.

2. A Privacy-Preserving On-Device Architecture: The project demonstrates a viable architecture for deploying a 127 MB transformer model on entry-level to mid-range Android devices without requiring cloud connectivity. The combination of lazy loading, singleton management, and largeHeap allocation provides a replicable pattern for other on-device ML applications in resource-constrained environments.

3. Ghana-Specific Fraud Pattern Rules: The 16 regex patterns in SmishGuard's rule engine encode domain knowledge specific to the Ghanaian mobile money ecosystem, including GhanaPost delivery impersonation, NLA/Afriluck lottery scams, USSD shortcode patterns, and WhatsApp-redirected social engineering. This pattern library represents a codified resource that can be extended and refined as new fraud vectors emerge.

4. Empirical Stakeholder Insights: The survey and interview data provide empirical evidence of the SMS fraud landscape in Ghana from the user's perspective, including prevalence data, current coping strategies, privacy attitudes, and feature preferences. These findings can inform the design of future mobile security interventions targeting similar populations.


6.4 Limitations

Several limitations must be acknowledged:

6.4.1 Dataset Constraints

The training dataset, while supplemented with Ghana-specific samples, may not fully represent the diversity of SMS fraud patterns in circulation. Smishing campaigns evolve rapidly, and the model's ability to detect novel patterns is limited by the training data's coverage. Continuous retraining with updated datasets would be needed to maintain detection effectiveness over time.

6.4.2 Language Limitations

The DistilBERT model is trained on English-language text, which limits its effectiveness with messages written in Ghanaian local languages (Twi, Ga, Ewe, Hausa) or Pidgin English. While many SMS fraud messages in Ghana are written in English, the inability to handle local-language variations represents a gap in coverage. The rule-based layer partially compensates by detecting language-agnostic patterns (URLs, phone numbers, USSD codes), but semantic analysis of non-English content is not supported.

6.4.3 Model Size

The 127 MB TFLite model, while manageable on the target device, is substantial for a mobile application. Users with limited storage space or very low-end devices (1–2 GB RAM) may find the application impractical. Techniques such as knowledge distillation, pruning, or int8 quantisation could reduce the model footprint at the cost of potential accuracy reduction.

6.4.4 Platform Limitation

SmishGuard is available only for Android. Apple's iOS restricts third-party access to SMS content, making a direct port impossible without jailbreaking or enterprise provisioning. This limits the application's reach to Android users.

6.4.5 Static Model

The deployed model is static — it does not learn from user feedback or adapt to new fraud patterns after deployment. If new smishing campaigns emerge that differ substantially from the training data, the application cannot improve its detection without a manual model update distributed via an app update.

6.4.6 Evaluation Scale

The usability testing sample of five participants, while standard for formative usability evaluation (Nielsen, 2000), limits the generalisability of usability findings. A larger-scale deployment study would provide more robust evidence of user satisfaction and behavioural impact.


6.5 Future Work

The following directions are identified for future development:

6.5.1 Federated Learning for Model Updates

Federated learning (McMahan et al., 2017) would enable the model to improve over time by aggregating learning from user devices without centralising SMS data. Each device would train locally on newly encountered messages and transmit only model gradients (not data) to a central server for aggregation. This approach would preserve the on-device privacy guarantee while enabling the model to adapt to emerging fraud patterns.

6.5.2 Multilingual Support

Expanding the model to handle Twi, Pidgin English, and other Ghanaian languages could significantly improve detection coverage. Multilingual transformer models such as mBERT or XLM-RoBERTa (Conneau et al., 2020) could serve as alternatives to the English-only DistilBERT model. This would require the collection and labelling of local-language SMS samples for training.

6.5.3 Model Compression

Applying knowledge distillation (Hinton et al., 2015) to train a smaller student model from the current DistilBERT teacher, or applying structured pruning and int8 quantisation, could reduce the model size from 127 MB to under 30 MB, substantially improving compatibility with lower-end devices.

6.5.4 User Reporting and Feedback Loop

Implementing a mechanism for users to report false positives and false negatives would provide two benefits: immediate correction of individual misclassifications for the user and, when combined with federated learning, a feedback signal for model improvement. Reports could be stored locally and optionally transmitted (with explicit user consent) for aggregation.

6.5.5 Integration with Telecommunications Providers

Collaboration with MTN, Telecel, and AirtelTigo could enable the integration of SmishGuard's detection capabilities with network-level filtering systems, providing a layered defence where network filters catch bulk SMS spam and SmishGuard catches targeted smishing that bypasses network defences.

6.5.6 Cross-Platform Deployment

While iOS restrictions prevent direct SMS access, a modified version could integrate with iMessage extensions or leverage screen-reading accessibility services (where permitted) to provide some level of detection capability on Apple devices.


6.6 Final Remarks

SmishGuard demonstrates that effective, privacy-preserving SMS fraud detection is achievable on mainstream Android hardware using a combination of transformer-based machine learning, domain-specific rules, and thoughtful architectural design. The application addresses a genuine and pressing need for Ghanaian mobile money users, who face persistent exposure to SMS-based fraud yet lack accessible tools designed for their specific context. By processing all data on-device, providing human-readable explanations, and operating transparently as a background service, SmishGuard offers a model for how mobile security applications can be both technically effective and trustworthy to the users they aim to protect.
